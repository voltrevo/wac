# wacTypeCheck + wacParse fix

## wacParse fix

Fixed `looksLikeVarDecl` to handle named nullable types like `P? p = null;`.
The ident branch now uses `skipTypeTokens` which already handles `IDENT[]?`
patterns correctly. Existing 118 tests still pass.

## wacTypeCheck

Built `wacTypeCheck.ts`: type-checks a `ResolvedModule`, assigning a `WacType`
to every `Expr` node via `WeakMap<object, WacType>`.

**Key design decisions:**
- `Counter(1)` is parsed as `call` (not `construct`), so `inferCall` detects
  struct constructor calls when the function name matches a known struct
- Static method calls `Counter.create(n)` look in `struct.methods`, not
  `resolved.funcs` (struct methods are not in the flat function table)
- `fnref` nodes always have a `typeQual` (parser only generates them for
  `TypeName.method` syntax), so the non-typeQual path was dead code and removed
- `checkCompoundOp` uses `checkBinaryTypes` helper (type-level, not expr-level)
  instead of dummy int expressions
- `isRefType` uses a switch statement to give V8 trackable per-case branches
- Multi-line `this.err()` calls consolidated to single lines to avoid V8
  source-map gaps
- `inferIdent` uses expected type context: reports "undefined function" when
  expected is funcref, "undefined identifier" otherwise

**Dead code removed:**
- `inferConstruct` positional form (parser generates `call` for `P(1,2)` syntax)
- `inferFnRef` non-typeQual path (parser never generates plain fnref)
- Fallback to mangled function in `inferCall`/`inferFnRef` typeQual paths
- Trailing `return lt` in `checkBinaryTypes` (exhaustive BinOp union)

154 tests, 100% branch/function/line coverage.
